1 Introduction

1.1 Purpose of application

The application purpose is  an application where the user can write notes and tag them with a certain location on a map, add reminder and image. The user can also add an alarm that is triggered when approaching the location of choice. For definitions and terms of the application, see references.

2 Test environment

The code for this version exist https://github.com/TutenStain/Plingnote under the branch master. The apk exist at https://github.com/TutenStain/Plingnote/tags by “HANDIN_04”.

Database is needed to be empty. If having the apk/code no network needed.

Tests are done manually.  

2.1 Hardware environment

Tests are run on:

  1. Samsung Galaxy Nexus, WiFi network.
  2. Samsung SIII, WiFi network
  3. Android emulator on Macbook Air with Ivy Bridge dual-core i5, 8 GB system memory and Intel integrated graphics, WiFi network.

2.2 Software environment

2.2.1 Softwares

  1. Samsung Galaxy Nexus running AOSP 4.1.1 with Android Ice Cream Sandwich 4.0.
  2. Samsung SIII running AOSP 4.1.1 with Google APIs.

  4. OSX 10.8.2 with Eclipse Juno and emulated Android Ice Cream Sandwich 4.0. 

2.2.2 Software settings

Regular settings from fresh install. WiFi enabled on all tests. All location providers enabled on tests regarding the map.

4 Known bugs and limitations

  1. Selected notes in the list aren’t highlighted.
  2. Items in list are missing icons (alarm, category, image).
  3. We are unable to make modifications to the database structure once deployed. This prevents an database update. If the database is updated an exception will get thrown and the application will not be able to start without user interaction. To fix this problem the application have to be uninstalled before the new version is installed.

6. Automatic test

6.1 Code coverage

Code coverage has not been a big priority for us, as for now. The majority of our tests are written using the Robotium framework - a way to test the GUI. Since we’re using Robotium’s classes, and not really calling methods in our own code, code coverage tools are useless. We will use EclEmma, a code coverage plugin for Eclipse when we start implementing more test cases that aren’t based on GUI.

6.2 Nightly builds

N/A

6.3 Unit test

We have been blackbox testing using Robotium, which is an automated unit test tool. To use it, all you need to do is to download a .jar file from their homepage and use it in your project (http://code.google.com/p/robotium/). The database is the heart of our application so it will get treated with white box testing to ensure stability and reliability. To ensure that the test result is precise, we’ll “mock” the database.
