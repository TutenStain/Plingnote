1 Introduction

This section gives a brief overview of the project.

1.1 Purpose of application

The project aims to create an application where the user can write notes and tag them with a certain location on a map. The user can also add an alarm that is triggered when approaching the location of choice. For definitions and terms of the application, see references.

1.2 General characteristics of application

Plingnote will be a standalone Android application which will be distributed via Google Play.

The application will have two tabs, displaying saved notes in two different ways - on a map or in a regular list. It’s possible to add a note directly to the map, but the location can also be added (or changed) at a later point. The main reason why a user would want to tag his/her notes with a location is because of the built-in alarm, which can be set to notify the user when arriving to a.

1.3 Scope of application

This main feature of the application is to view and create notes. Even if this project relies heavily on text, no custom text styles will be supported. Database is used for all data storage, but no backup will be performed to secure the user’s notes. No native tabled support will implemented.

1.4 Objectives and success criteria of the project

For the project to be called successful we want to implement the following features:

  1. Add notes - the user should be able to add notes effortlessly.
  2. Add notes via the map - the user should be able to add notes going from the map view. This way the note will automatically be tagged with the selected location.
  3. Map view - a place to see the tagged notes on a map.
  4. List view - a traditional list of the notes, ordered by date of creation.
  5. Snotebar - placed at the bottom of the screen when the user is editing a note, handling the note settings.
  1. Map - view or change the location tagged with the note.
  2. Alarm -  add and change alarm. From here it is also possible to set repetition time for the alarm, like very Mondays.
  3. Category - change the category for the note opened from presets like food or banking.
  4. Custom image - attach an image to the note from the gallery. The user can also shoot a new picture using the camera.

1.5 Definitions, acronyms and abbreviations

  1. Java, platform independent programming language
  2. Database, an organized collection of data
  3. APK, the format Android system uses to install applications. APK stands for Android Application Package

2 Requirements

In this section we specify all requirements.

2.1 Functional requirements

2.1.1 - Zoom map

Scenario: Viewing notes in map view.                                        

Trigger: The user is pinching on the map.

Precondition: The map tab is active

Basic path: When the user zooms in or out the map the zoom level will be changed. New annotations will be loaded and shown when the user is not zooming any longer.

Exceptional path: If no new annotations can be loaded due to any connection issues, see requirements.

Status on completion: The view will be centered around a new location, specified by the user, and the annotations passing the current filters will be shown.                                        

2.1.2 - View map                                                

Scenario: View map

Trigger: Starting the app or pressing the map tab.

Precondition:

Basic path: The user starts the application. When the app launches, a map with nearby annotations will be shown.

Status on completion: The map activity and the list activity should be started and loaded, and the map activity should be shown        

2.1.XXX - Zoom to current position                                                

Scenario: Zoom to current position

Trigger: Presses the find current position on the map in the map tab. The button is located on the top right corner

Precondition: A data bearer connection active.

Basic path: When the location is found or approximated the map will be zoomed in and the annotations will be shown.

Status on completion: The map will get zoomed in to current position.                        

2.1.3 - Add note without pre-set location

Scenario: Viewing notes in either map view or list view.                                                

Trigger: Press the new note button in the upper right corner.

Precondition: None

Basic path: A new view, where the user can write a note, is displayed.

2.1.4 - Add note with pre-set location

Scenario: Viewing notes in map view.                                                

Trigger: Holding finger at the spot where the user wants the note to be added.

Precondition: An active data bearer connection.

Basic path: A new view, addnote where the user can write a note, is displayed. The note’s position is saved.

Exceptional path: A note is already placed at the given position. The user is then asked to edit or remove the current note, or add a new one at the same place.

2.1.5 - Search

Scenario: Viewing notes in either map view or list view.                                                

Trigger: Presses the search icon at the upper actionbar.

Precondition: None

Basic path: While the user is typing, a list is filled with notes that matches the written text string - by title or by content.

2.1.XXX - Search Suggestions

Scenario: User is searching.                                                

Trigger: Presses the search icon at the upper actionbar.

Precondition: The search field have focus and the user have entered at least 3 characters. The user have previously searched for notes starting with the same letters as currently typed.

Basic path: While the user is typing, a list is filled with search suggestions that matches the written text string will be show. This is based on previously entered search queries.

Exceptional Path: User have previously searched for the same thing that the user is searching for now but no suggestion appear. This is because the user previously just have cleared the search history.

2.1.XXX - Clear Search History

Scenario: User is changing or exploring applications settings.

Trigger: Presses the clear search history button and accepts the prompt to really clear the search history.

Precondition: The user is in the settings panel.
Basic path: All search history will be cleared.

2.1.XXX - Send Feedback

Scenario: User wants to send feedback to the developers.                                                

Trigger: Presses the send feedback button in the settings screen.

Precondition: Is in the settings view.

Basic path: The default email client is opened and the user enters comments, request or error reports that the user will then send to the developers.

Exceptional Path: User decides that he/she does not want to send the email. Presses back and the message is saved as a draft (based on the default email client settings). Ends up at the homescreen and launches the application (Plingnote) again.

2.1.XXX - View Terms of Use

Scenario: User wants to view his/her rights.                                        

Trigger: Presses the terms of use button.

Precondition: Is in the settings view.

Basic path: Is presented with the terms of use. Dismisses the dialogue with the okay or the devices back button.

2.1.XXX - View Open Source Licenses

Scenario: User wants to view the open source licenses that the application is using.        

Trigger: Presses the open source licenses button

Precondition: Is in the settings view.

Basic path: Is presented with the open source licenses that the application is using. Dismisses the dialogue with the okay or the devices back button.

2.1.XXX - View application information and version

Scenario: User wants to view the applications information, version history and changelog..        

Trigger: Presses the Plingnote button.

Precondition: Is in the settings view.

Basic path: Is presented with application information such as version and changelog.s.

2.1.XXX - Search Suggestions

Precondition: The search field have focus and the user have entered at least 3 characters. The user have previously searched for notes starting with the same letters as currently typed.

Basic path: While the user is typing, a list is filled with search suggestions that matches the written text string will be show. This is based on previously entered search queries.

Exceptional Path: User have previously searched for the same thing that the user is searching for now but no suggestion appear. This is because the user previously just have cleared the search history.

2.1.XXX - Change Settings

Trigger: Presses the overflow menu and then selects the settings option

Precondition: The application is started.

Basic path: User changes settings.

2.1.6 - Scroll map

Scenario: View map

Trigger: Scrolling or swiping the map

Precondition: The map activity is active

Basic path: When the user scrolls or swipes on the map, new notes will be shown accordingly.

Exceptional path: If no new annotations can be loaded due to any connection issues, see requirements.

Status on completion: The view will be centered around a new location, specified by the user, and the annotations passing the current filters will be shown.

2.1.7 - Change note view

Scenario: Viewing notes in map view or list view.

Trigger: Pressing the desired view icon (map or list) or starts swiping from the left or right edge of the screen.

Precondition: The application is started

Basic path: The way the notes are displayed is changed.

Exceptional path: The user is already in the selected view - nothing happens.

2.1.8 - Show contextual action bar

Scenario: Viewing notes in list view.

Trigger: Pressing a finger on a note until the device vibrates and the menu appears.

Precondition: There is at least one note in the list.

Basic path: A menu is displayed at the top of the screen. The user can then select more items and perform an action on them.

2.1.XXXX - Delete notes from list

Scenario: The contextual bar is visible at the top of the screen, which means that at least one note is selected.

Trigger: User is pressing the “Trash” icon in the top menu.

Precondition: None

Basic path: The selected notes are removed from the list.

2.1.9 - Edit / view note

Scenario: User want to view or edit an already

Trigger: Opens a note from either the maview or listview.

Precondition: The application is running and the user is either in the mapview or listview.

Basic path: Opens a note and view or edits it before leaving the view or the application.
Status on completion: The user reads the note and if its edited it will get saved and updated.

2.1.9 - Write note headline

Scenario: Write a headline for a new note.

Trigger: Presses the add note in the actionbar.

Precondition: The note activity is active.

Includes: Write note

Basic path: The user enters a text using the onscreen keyboard that pops up. Once the headline is entered the user presses enter (newline) and continues to write the actual note.

Status on completion: The headline is saved and can be later edited by opening up the note in either the listview or mapview.

2.1.10 - Change note settings

Scenario: Change or set note settings.

Trigger: Touch the icons on the note settings part of the screen.

Precondition: The setting you want to change is set to something and you need to be in the editing view.

Basic path: If the user touches one of the icons in the lower part of the screen the user will be able to set or change the settings.

Exceptional path: If the user want to  reset a setting, hold finger on the icon.

Status on completion:  The setting icon is colourful and showing the users choice.

2.1.11 - View note settings

Scenario: The user wants to look at settings.

Trigger: Nothing, scroll on note view or scroll on the note settings.

Precondition: The note activity is active.

Basic path: If the user has chosen a setting that setting icon will be colourful and with the necessary information. Otherwise the icon is grey. The user can scroll left and right to see more settings.

2.1.14 - Set picture

Scenario: Attach a picture to the note.

Includes: View note settings

Trigger: Touch the add picture icon on the lower part of the screen.

Precondition: The note activity is active.

Basic path: The user views and and interacts with the Android standard gallery view selecting a picture to attach to the note. The user can also open up the standard default camera application to take a picture and select it.

Status on completion: The picture will be set and saved. The picture will get attached to the note for later viewing.

2.1.15 - Set category

Scenario: Set the category for the note like food or banking.

Includes: View note settings

Trigger: Touch the category icon on the lower part of the screen.

Precondition: The note activity is active.

Basic path: The user views and and interacts with the Android standard gallery view selecting the most appropriate category icon.

Status on completion: The category will be set and saved. The category icon will be show on the map if the user have selected a specific location for the note. If category is selected the icon will always be shown in the list view instead of the standard dot (*).

2.1.16 - Set location

Scenario: Set a location on the map

Includes: View note settings

Trigger: Touch the alarm icon on the lower part of the screen.

Precondition: The note activity is active.

Basic path: The user views and or interacts with the map like pan, zoom and scroll. Clicking and holding on a specific location changes the marker to that position.

Status on completion: The position will be set and saved. If the user nears the position within a radius the app will trigger the note notification.

2.1.17 - Add alarm/notification

Scenario: Set an alarm/notification

Includes: View note settings

Trigger: Touch the alarm icon on the lower part of the screen.

Precondition: The note activity is active

Basic path: User sets and configures the alarm. Changes date, time and sets if the alarm should repeat.

Status on completion: Alarm will be fired on according to the configured time and date.

2.2 Non-functional requirements

2.2.1 Usability

Usability has a high priority. The application must be effective, easy, intuitive and fun to use.

2.2.2 Reliability

The applications should not be prone to crashes, data loss or corruption.

2.2.3 Performance

Performance have a high priority. Our applications map view response/loading time is heavily depending on the Internet connection currently active. Any action initiated by the user, excluding viewing the map, should definitely not exceed one second.. It should be easy and fast to write notes.

2.2.4 Supportability

The application will be platform dependent on the Android mobile operation system. All hosts must have Android version >= 4.0 (Ice Cream Sandwich) installed on their phone or tabled to be able to run the application.

2.2.5 Implementation

This is a open source project under GNU General Public License version 3 so all documentation, source code, resources, etc which can be downloaded from our git repository can be delivered by any medium. Git repository: https://github.com/TutenStain/Plingnote

2.2.6 Packaging and installation

Plingnote will be delivered as an installable APK file.

2.2.7 Legal

Our license will be the GNU General Public License version 3. We find this license appropriate as free software easily can reach out to a large amount of people. In addition, should any other students wish to study or improve our code, they will have that opportunity.

One of the reasons why we chose GPLv3 instead of other open source licenses is its clarity regarding the user’s freedom. The terms of GPLv3 also state very clearly that once software is licensed under its terms, redistributed versions of that software will be licensed under those terms as well.

Another reason to why we are into using GPLv3 is that we think that this application has a great idea behind it and we want to see other people taking it to higher levels. Since this is such a great idea it’s most likely that there will be a price taken for the application and we have the thoughts that this should be a free app that anyone can improve.

Open source licenses can be found in the application settings view.
