1 Introduction

1.1 Purpose of application

Plingnote is an application where the user can write notes and tag them with a certain location on a map. The user can also add an alarm that is triggered when approaching the location of choice.

1.2 General characteristics of application

Plingnote will be a standalone Android application which will be distributed via Google Play.

The application will have three tabs, displaying saved notes in different ways - on a map, in a regular list and in a grid. It’s possible to add a note directly to the map, but the location can also be added (or changed) at a later point.

2 Test environment

The code for this version is available at https://github.com/TutenStain/Plingnote, in the master branch. APK can be found in HANDIN_05 at https://github.com/TutenStain/Plingnote/tags. For the program to launch correctly, the database needs to be empty. No network is needed. Tests are done manually.  

2.1 Hardware environment

Tests are run on:

  1. Samsung Galaxy Nexus, WiFi network.
  2. Samsung SIII, WiFi network
  3. Android emulator on Macbook Air with Ivy Bridge dual-core i5, 8 GB system memory and Intel integrated graphics, WiFi network.

2.2 Software environment

2.2.1 Softwares

  1. Samsung Galaxy Nexus running AOSP 4.1.1 with Android Ice Cream Sandwich 4.0.
  2. Samsung SIII running AOSP 4.1.2 with Google APIs.
  3. Samsung SIII running Samsung TouchWiz 4.0.4 with Android Ice Cream Sandwich 4.0.
  4. OSX 10.8.2 with Eclipse Juno and emulated Android Ice Cream Sandwich 4.0.

2.2.2 Software settings

Regular settings from fresh install. WiFi enabled on all tests. All location providers enabled on tests regarding the map.

3 System information

3.1 System version

Plingnote v0.2, release 2.

4 Known bugs and limitations

  1. When rotating the screen while items are selected in the list view, the contextual action menu disappears and the items are unselected.
  2. Search bar is closed when rotating the device.
  3. If you select an image on a note, they will not appear in the gridview.
  4. The database does not back up data when upgraded.

6. Automatic test

6.1 Code coverage

The majority of our current tests are written using the Robotium framework - a way to test the GUI of the application. Since we’re using Robotium’s classes, and not really calling methods in our own code, code coverage tools are useless. We do think that having a high code coverage is of great value when developing applications, and we’ll aim for as high percentage covered as possible.

6.2 Nightly builds

N/A

6.3 Unit test

We have been blackbox testing using Robotium, which is an automated unit test tool. To use it, all you need to do is to download a .jar file from their homepage and use it in your project (http://code.google.com/p/robotium/). The database is the heart of our application so it will get treated with white box testing to ensure stability and reliability. To ensure that the test result is precise, we’ll “mock” the database.
